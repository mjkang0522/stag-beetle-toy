# 에이전트 코딩 규칙

이 문서는 이 폴더에서 작업하는 코딩 에이전트가 따라야 할 규칙이다. 이 프로젝트는 정적 HTML/CSS/JavaScript 기반의 작은 브라우저 토이이므로, 단순함과 애셋 안정성을 최우선으로 둔다.

## 프로젝트 기본 원칙

- 빌드 도구, 프레임워크, 패키지 매니저는 명확한 필요가 생기기 전까지 추가하지 않는다.
- 기존 구조인 `index.html`, `css/style.css`, `js/main.js`, `assets/`를 유지한다.
- 작은 기능은 먼저 `js/main.js` 안에서 일관되게 확장하고, 파일 분리는 복잡도가 실제로 커졌을 때만 한다.
- 사용자-facing 텍스트, 문서, 주석은 현재 프로젝트 분위기에 맞춰 한국어를 우선 사용한다.
- 기존 코드 스타일을 따른다: 4칸 들여쓰기, 세미콜론 사용, 명확한 함수명, 과한 축약 금지.

## HTML 규칙

- `index.html`은 화면 구조만 담당한다.
- 새 UI를 추가할 때는 의미 있는 요소를 우선 사용한다. 클릭 가능한 것은 가능하면 `button`을 사용한다.
- 이미지에는 적절한 `alt`를 유지한다.
- 스크립트와 스타일 경로는 현재처럼 루트 기준 상대 경로를 사용한다.

## CSS 규칙

- `#game_screen`은 전체 화면 배경 컨테이너로 유지한다.
- `body`의 스크롤 방지와 여백 제거는 유지한다.
- 사슴벌레 입력 안정성을 위해 `touch-action`, `user-select`, 이미지 드래그 방지 설정을 함부로 제거하지 않는다.
- 모바일 화면을 고려해 새 크기값은 가능하면 `clamp()`, `%`, `vw`, `vh` 같은 반응형 단위를 검토한다.
- 시각 요소를 추가할 때 기존 배경과 사슴벌레 애셋을 가리지 않도록 z-index와 위치를 명확히 관리한다.

## JavaScript 상태 규칙

- 사슴벌레의 현재 상태는 `beetle` 객체를 단일 출처로 삼는다.
- 화면 반영은 가능하면 `updateBeetleView()`를 통해 처리한다.
- 새 상태를 추가할 때는 `beetleFrames`와 `animationSettings`에 먼저 정의한다.
- 프레임 번호는 현재 코드처럼 1부터 시작한다. 배열 인덱스로 직접 바꾸지 않는다.
- 사용자를 통한 즉시 반응 상태는 시작 시 `clearTimers()`를 호출해 기존 idle, walk, frame timer를 정리한다.
- 타이머를 추가하면 반드시 정리 경로를 함께 만든다. 정리 후 변수는 `null`로 되돌린다.
- 걷기 이동은 현재처럼 퍼센트 좌표와 `requestAnimationFrame` 보간을 기준으로 한다.
- 방향 처리 규칙을 유지한다: 원본 사슴벌레 이미지는 왼쪽을 보고 있으므로 오른쪽 이동 시 `scaleX(-1)`을 사용한다.

## 애니메이션 추가 규칙

- 애니메이션은 "상태 이름", "프레임 목록", "프레임 순서", "프레임 지연 시간"을 분리해서 정의한다.
- 반복 애니메이션과 일회성 반응 애니메이션을 섞을 때는 종료 콜백에서 다음 상태를 명확히 호출한다.
- 새 인터랙션은 기존 흐름을 깨지 않게 `idle -> action -> idle` 또는 `idle/walk -> reaction -> idle` 구조를 우선 검토한다.
- 동시에 두 개 이상의 `setInterval`이 같은 이미지를 바꾸지 않게 한다.

## 애셋 규칙

- 사슴벌레 프레임 PNG는 현재 `512 x 512` 기준이다. 새 프레임도 가능하면 같은 캔버스 크기와 정렬을 유지한다.
- 기존 파일명을 임의로 바꾸지 않는다. 특히 `beetle_eat_oepn_*.png`는 오타처럼 보여도 실제 파일명이므로, 수정하려면 코드와 파일명을 함께 마이그레이션한다.
- 새 사슴벌레 이미지는 `assets/beetle/`에 둔다.
- 새 배경은 `assets/bg/`에 둔다.
- 새 효과음은 `assets/sound/`, 새 UI 이미지는 `assets/ui/`에 둔다.
- 사용하지 않는 애셋을 삭제하기 전에는 향후 기능 후보인지 먼저 확인한다.

## 접근성 및 입력 규칙

- 포인터 이벤트를 기본 입력으로 사용하되, 새 UI에는 키보드 조작도 고려한다.
- 터치 기기에서 의도치 않은 확대/스크롤/드래그가 생기지 않게 확인한다.
- 사슴벌레 자체에 더 많은 상호작용을 붙이면 `aria-label`, `tabindex`, 키 입력 처리를 검토한다.

## 검증 규칙

- 코드 변경 후 최소한 브라우저에서 `index.html` 또는 로컬 서버 실행 화면을 확인한다.
- 로컬 서버가 필요하면 프로젝트 루트에서 `python -m http.server 8000`을 사용한다.
- 콘솔 에러가 없는지 확인한다.
- 사슴벌레가 idle, walk, touch, happy 상태를 정상적으로 오가는지 확인한다.
- 새 애셋을 연결했다면 경로 대소문자와 파일명 오타를 확인한다.

## 문서화 규칙

- 프로젝트 구조나 주요 상태 흐름을 바꾸면 `docs/PROJECT_ANALYSIS.md`도 함께 업데이트한다.
- 에이전트 작업 방식 자체가 바뀌면 이 `AGENTS.md`를 업데이트한다.
- 구현 의도를 설명하는 주석은 환영하지만, 코드가 이미 말하는 내용을 반복하는 주석은 피한다.

## 피해야 할 일

- 단순 기능 추가를 위해 React, Vue, Vite 같은 도구를 바로 도입하지 않는다.
- 기존 애셋 파일명을 단독으로 변경하지 않는다.
- 상태별 타이머 정리 없이 새 인터랙션을 추가하지 않는다.
- 전역 상태를 여러 곳에 흩뜨리지 않는다.
- 화면 크기와 터치 입력을 확인하지 않은 채 고정 좌표 UI를 추가하지 않는다.
