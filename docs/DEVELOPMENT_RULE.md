# 개발 원칙

이 문서는 앞으로 `우리집 사슴벌레` 프로젝트를 수정할 때 항상 지켜야 할 기준이다.

## 핵심 원칙

- 항상 실행 가능한 상태를 유지한다.
- 기능은 한 번에 하나씩 구현한다.
- 기존 코드를 최대한 유지한다.
- 가독성과 유지보수를 우선한다.
- MVP를 우선한다.
- 새로운 기능을 추가하기 전에 현재 기능을 더 단순하게 만들 수 있는지 먼저 검토한다.
- 모든 구현은 초보자가 이해하기 쉬운 구조를 우선한다.

## 코드 원칙

- 빌드 도구, 프레임워크, 패키지 매니저는 명확한 필요가 생기기 전까지 추가하지 않는다.
- 기존 구조인 `index.html`, `css/style.css`, `js/main.js`, `assets/`를 유지한다.
- 작은 기능은 먼저 `js/main.js` 안에서 일관되게 확장한다.
- 파일 분리는 코드가 실제로 읽기 어려워졌을 때만 한다.
- 사용자-facing 텍스트, 문서, 주석은 한국어를 우선한다.
- 4칸 들여쓰기와 세미콜론 사용을 유지한다.
- 함수명은 동작이 드러나도록 명확하게 쓴다.

## 상태 관리 원칙

- 사슴벌레의 현재 상태는 `beetle` 객체를 단일 출처로 삼는다.
- 화면 반영은 가능하면 `updateBeetleView()`를 통해 처리한다.
- 새 상태를 추가할 때는 `beetleFrames`와 `animationSettings`를 먼저 검토한다.
- 프레임 번호는 1부터 시작한다.
- 즉시 반응 상태를 시작할 때는 먼저 `clearTimers()`를 호출한다.
- 타이머를 추가하면 반드시 정리 경로를 함께 만든다.
- 정리한 타이머 변수는 `null`로 되돌린다.
- 걷기 이동은 퍼센트 좌표와 `requestAnimationFrame` 보간을 기준으로 유지한다.

## 애셋 원칙

- 사슴벌레 프레임 PNG는 가능하면 `512 x 512` 캔버스와 정렬을 유지한다.
- 새 사슴벌레 이미지는 `assets/beetle/`에 둔다.
- 새 배경은 `assets/bg/`에 둔다.
- 새 효과음은 `assets/sound/`에 둔다.
- 새 UI 이미지는 `assets/ui/`에 둔다.
- 기존 파일명은 코드와 함께 마이그레이션할 때만 변경한다.
- `beetle_eat_oepn_*.png` 파일명은 현재 실제 애셋명이므로 단독으로 고치지 않는다.

## 검증 원칙

- 변경 후 브라우저에서 실행 화면을 확인한다.
- 로컬 서버가 필요하면 프로젝트 루트에서 `python -m http.server 8000`을 사용한다.
- 콘솔 에러가 없는지 확인한다.
- idle, walk, touch, happy 흐름이 정상인지 확인한다.
- 새 애셋을 연결했다면 경로 대소문자와 파일명 오타를 확인한다.
- 모바일 터치에서 의도치 않은 스크롤, 확대, 이미지 드래그가 없는지 확인한다.

## 문서 업데이트 원칙

- 현재 기능, 상태, 리소스가 바뀌면 `docs/PROJECT_STATUS.md`를 업데이트한다.
- 애니메이션 프레임, 순서, 지연 시간, 상태 흐름이 바뀌면 `docs/ANIMATION_GUIDE.md`를 업데이트한다.
- 완료한 작업은 `docs/CHANGELOG.md`에 날짜와 함께 추가한다.
- 앞으로 할 일이 생기거나 우선순위가 바뀌면 `docs/TODO.md`를 업데이트한다.
- 개발 규칙 자체가 바뀌면 `docs/DEVELOPMENT_RULE.md`와 필요한 경우 `AGENTS.md`를 함께 업데이트한다.
- GitHub 첫 화면에 보여야 하는 소개, 실행 방법, 전체 진행 상황이 바뀌면 `README.md`를 업데이트한다.
- 구조 분석이나 문서 배치가 크게 바뀌면 `docs/PROJECT_ANALYSIS.md`를 업데이트한다.
